项目定位与主链路

Overview

聚合 Polymarket + Kalshi 两家预测市场,归一化后做跨平台配对,再用多 Agent 分析层产出交易判断,最后用复刻 Bloomberg 风格的终端 UI 呈现。仅只读公开行情,不下单。

[接入] Polymarket WS / Kalshi REST
    ↓ normalize(纯函数翻译)
[持久化] Postgres  markets · ticks(时序)· signals · market_pairs · agent_runs
    ↓
[配对] curated + Jaccard 预筛 + LLM 裁判 ──► market_pairs
    ↓
[洞察] ProbTracker · Screener · AnomalyDetector ──► signals
    ↓
[决策] Agent 流水线:分析师 → 多空辩论 → 交易员/风控 ──► AgentVerdict
    ↓
[体验] board 聚合(一事件一行)· SSE 实时推流 · 终端 UI

全 TypeScript / Next.js(App Router)。instrumentation.ts 在服务启动时拉起常驻摄取服务,通过内存事件总线把更新经 SSE 推给前端。两个核心开关:DATA_SOURCE(seed/live)与 AGENT_ENGINE(deterministic/llm)。

归一化:接入层的翻译墙

Normalize

核心就一个文件 connectors/normalize.ts —— 纯函数、零 I/O。把两家五花八门的原始 JSON,一次性压成统一领域类型 MarketMeta / MarketTick,下游不再关心数据来自哪家。

它消化的四类「脏」

问题表现处理
字段名不同Poly conditionId/question vs Kalshi ticker/title查表搬运到统一字段
类型不干净数组被当字符串返回、数字混字符串parseMaybeJsonArray / toNumber
单位不一致Kalshi 价格时而美分(0–100)时而美元(0–1)启发式 n > 1.5 ? n/100 : n
YES 口径不同Poly 取 outcomePrices[0];Kalshi 取买卖中点统一成 yesProb 0..1,clampProb 夹边界

YES 概率的口径统一(最关键)

  • Polymarket:REST 直接取 outcomePrices[0];实时 WS 从订单簿算 best bid/ask 中点。
  • Kalshi:取 (yes_bid + yes_ask)/2,单边则用单边,都没有退回 last_price。

两边都收敛到「中点」口径,是后面「跨平台价差直接相减」能成立的前提。

connector 负责取数 + 连接管理(WS 重连 / 轮询 429 退避);normalize 负责翻译。职责分离,所以归一化能用 __fixtures__ 真实样本纯函数单测。

seed vs live 双模式

Seed vs Live

看板里那 8 个事件,是硬编码的 Demo 种子,不是抓取归一化后剩下的。

  • seed 来自 seed/prototype.ts 的 PROTOTYPE_BASE —— 正好 8 条,逐字从原型 HTML 搬来,概率/成交全是写死常量。getMarketViews() 第一行就短路返回,事件一个不增减。作用:让整条链路离线、零依赖、可复现地端到端跑通,且与原型 UI 像素一致。
  • live 才真去抓:Poly/Kalshi 各拉若干 → 过滤 → 配对。列表行数 = 配对成功的事件对数,由「拉取量 × 配对成功率」决定,通常不等于 8。

live 链路:抓取 → 配对 → 落表

Live Pipeline

入口 instrumentation.ts → startIngest() → live 时进 startLive()(ingest/worker.ts)。

  1. 并行抓取:Poly 按 24h 成交倒序拉流动性最好的;Kalshi 过量抓取后滤掉多元市场(KXMVE*)、按成交排序取 top N。各自 .catch(()=>[]) 互不阻断。
  2. 落市场表:upsertMarkets 按 (source, marketId) 幂等。
  3. 跨平台配对:runMatching → matchMarkets,结果 upsertPairs 落 market_pairs。
  4. 实时 tick 流:Poly WS(心跳+指数退避重连)与 Kalshi 轮询(4s,429 退避)统一回调 onTick → insertTicks 落 append-only 时序表 + bus.emit。
  5. 周期重算:每 30s recomputeSignals 全表替换 signals,并刷新每对的 mergedYesProb。
  6. 聚合出表:GET /api/markets → getBoard(),for (p of pairs) 一对一行。

⚠ seed→live 第一个坑

curated.ts 的人工种子表在 live 下失效:它的 id 是合成的 poly-<id>/kalshi-<id>,和真实 conditionId/ticker 对不上。于是 live 配对 100% 依赖 LLM 裁判;没配 key 时只剩 Jaccard,极易全军覆没 → market_pairs 空 → 看板一行都没有。

跨平台配对:实体对齐

Matching

两家无公共标识,唯一桥是标题。这是经典的 Entity Resolution 难题,用「人工策展 → Jaccard 预筛 → LLM 裁判」三段式漏斗逼近。

[① curated] 已知事件写死,confidence=1 ── 优先
[② 预筛]  Poly×Kalshi 笛卡尔积 → Jaccard 相似度(阈值松到 0.04)→ 降序候选
[③ 裁判]  LLM 逐对判 same?/confidence(上限 60 次)→ ≥0.6 接受 → 一对一贪心占用
                                                    └──► market_pairs

分工:预筛只负责「别漏」(召回)、不负责「判准」;LLM 负责「判准」(精度)。阈值压到 0.04 就是为了让「BTC $150k」vs「Bitcoin $150,000」这种同义改写也能漏进裁判。

为什么 live 配对这么脆

结构性脆点

  • 召回层太弱:Jaccard 是纯词形重叠,对「自然语言 vs 代码标题」「数值实体」无能为力,零重叠就彻底漏掉。
  • curated 在 live 下是死的:本该最稳的一级失效,全部压力转嫁 LLM。
  • 截断 + 贪心:maxJudgements=60 可能让正确对没轮到就 break;一对一贪心不回溯,高相似但错误的对会「抢占」正确的对。
  • 降级即废:没 key 或 LLM 失败 → 退回纯 Jaccard。

一脆就「整个看板崩」

getBoard 只遍历 pairs —— 配对 0 对 → 列表 0 行,哪怕真实行情全在库里。套利信号(detectSpread)、mergedYesProb 也都只在 pair 上算。底层 buildAgentInput 其实支持「未配对单边展示」,但看板层没用上这个降级能力。

本质:把一个公认难题用「笛卡尔积 + Jaccard + LLM 单轮 + 贪心」一条脆链解决,又让产品主视图强依赖它的产出。任一环掉链 → 结果不是「少几行」而是「零行」。

judgePair Prompt 稳定性

Prompt

结论:方向对(点到了「同一结果口径」),但作为生产级裁判偏薄 —— 它能稳定产出「格式正确的判断」,却产不出「可靠的判断」。

做对的

  • 任务定义抓住要害:「同一结果口径」是配对真正的难点。
  • 输出契约有护栏:强制 JSON + response_format + extractJson 容错 + zod 校验,格式层基本不崩。

不够稳的(按严重度)

  • 口号无材料:嘴上要判结算口径,却只喂两个标题,category/endDate 一个没传。等于做缺条件的题。
  • 无 few-shot:宽松/严格的尺度全靠模型自由发挥,误配率不可控。
  • confidence 语义未定义:却被当成 ≥0.6 的硬阈值用,尺度还会漂移。
  • temperature 0.4 偏高:分类任务理应趋近 0;随机性还被 upsert 固化进数据。
  • 「重试」只换模型不纠错:仅异常触发,判错不会重判。
  • 缺「无法判断」出口:信息不足时被迫二选一,只能瞎猜。

最贵的失败不是程序崩,而是「默默给出错误配对且高 confidence」—— 会被当真套利信号展示,还被缓存复用。低成本加固:喂 category/endDate、写清规则与 confidence 刻度、加 2–3 个 few-shot、显式 temperature 0、放开低 confidence。全在 prompt 层,不动流水线。

分析框架:纯函数洞察层

Analysis

全是纯函数、零 I/O、可单测。把时序数据压成「信号 + 排序 + 一行快照」。工程最扎实的一层。

ProbTracker — 计量基础

probDelta(history, '1h'|'24h') 锚定最新 tick、回看窗口内最早一条,算 (last-first)×100 得 pts;volumeChangePct 算成交百分比变化。隐患:live tick 稀疏时窗口内找不到早点会返 0 → 信号「假性平静」。

Screener — 标的优选

维度权重含义
24h 成交0.35流动性/热度
累计成交0.25流动性代理
距结算时间0.20越近越高
近期活跃 |Δ24h|0.20概率在动

亮点 saturate(v,scale)=v/(v+scale) 边际递减归一化,把无界量稳健压到 0–1。注:rankMarkets 已写好且测了,但主链路尚未接入。

AnomalyDetector — 三类信号

信号触发severity
prob_jump 概率跳动|Δ24h| ≥ 5 ptsmin(1, |Δ|/15)
vol_spike 成交放量volChg ≥ 50%min(1, pct/200)
xplat_spread 套利价差|poly−kalshi| ≥ 4¢min(1, ¢/15)

套利信号死死绑在 pairs 上(脆点延伸)。口径瑕疵:prob_jump 文案写「2 小时内」,实算用 24h 窗口。

Agent 流水线:决策层

Agents

镜像 TradingAgents:分析师 → 多空辩论 → 交易员/风控,产出结构化 AgentVerdict。双引擎共用 AgentInput 输入契约,可互换、可对比。

引擎实现定位
deterministic原型决策树逐字移植,if-else 优先级 + 模板文案离线零成本可复现;Demo 内容源 + LLM 兜底基线
llm两段式调用:分析师段 → 交易员/风控段,zod 校验同一套规则下更丰富的中文论证

统一决策原则(写进 prompt,与决策树一字不差)

价差 ≥ 4¢          → 套利对冲  conf .78
24h↑ ≥ 5pts 且 <70% → 买入 YES   conf .64
24h↓ ≥ 5pts        → 买入 NO    conf .60
概率 ≥ 78%        → 观望       conf .52
其它              → 观望       conf .48

容错三层 — 产品永不硬失败

LLM 任何环节失败 → 回退确定性结果(但 engine 仍标 'llm');chatJSON 内部主模型 → 兜底模型重试;最外层确定性引擎兜底。

关键问题与加固清单

Issues

把全程发现的脆点与取舍汇总,按性价比给加固方向。

问题影响加固
看板只遍历 pairs配对 0 → 看板空白未配对市场单边展示(底层已支持),最小改动见效最快
curated 在 live 失效配对全压 LLM填真实 conditionId↔ticker,确定性命中热门事件
Jaccard 召回弱异构标题漏配预筛叠加 category / endDate / 数值实体信号
judgePair prompt 偏薄静默误配 + 高 confidence喂上下文 + few-shot + temp 0 + confidence 刻度
maxJudgements=60 截断正确对被截掉按类别分桶后裁,或调高上限
决策逻辑重复三处阈值改动易漂移抽成单一 source of truth
LLM 仅软遵守规则两引擎口径一致≠保证事后校验 LLM 输出是否违反阈值
计量口径瑕疵不确定性被「洗白」对齐 prob_jump 文案与窗口;liq 改真实簿深

总评:范式选型大多正确(漏斗式 hybrid matching、双引擎同规则、纯函数可测),工程质量上分析框架最扎实。最大的系统性风险集中在跨平台配对这一环:召回层与保底层在 live 下都偏弱,而产品主视图又强依赖它 —— 加固应优先投在「保底端(curated)」和「召回端(多信号预筛)」,而非 LLM 本身。
